1
订单保持「待审核」（状态不变）→ 写标记位 ReconnectDeadline = now+29min，三方详细 = 失败
2
该笔单移出该审核员列表 → 进「全部提现记录」可追溯；客户端立即弹全局重连弹窗 + 29min 倒计时（同步服务端 deadline）
3
复用现有能力：29min 内用户重连 KYC → 系统自动重提三方 → 订单转「已提交」→ 走分支 B
4
29min 到点仍未重连成功 → 后台自动取消 → 原路退回钱包，订单状态 = 已拒绝，原因 = 重连超时
⚠️ 重连后自动重提若再次失败，29min 不重置（以首次提交失败为准，一单一上限）

分支 B 通道出款提交成功 · 已入三方，进入匹配 / 验证窗口

1
订单转「已提交」，三方详细 = 等待回调；三方进入匹配窗口（如 60min）
2
窗口内 KYC 断连才弹客户端重连弹窗，倒计时 = 三方剩余窗口（非本地 29min）
3
三方成功回调 → 三方详细 = 成功 → 出款完成
4
三方失败 / 超时回调 → 系统判失败 → 原路退回钱包，订单状态 = 已拒绝

⚙️ 关键规则

规则说明
KYC重连剩余分支 A 固定 29 分钟，只存服务端、客户端同步显示；到点未重连成功 → 后台自动取消退回
自动重提复用现有「KYC 重连 → 自动重提三方」能力；本期只新增 29min 封顶。重提再失败 不重置倒计时（一单一上限）
分支 B 窗口倒计时 = 三方返回的剩余窗口（如 60min），由三方回调决定成败；客户端断连才弹
资金退回两分支统一原路退回钱包（解冻 + 写账变），由 V1 钱包执行（V3 发 MQ，带幂等键）
并发安全「重连自动重提」与「到点判失败」临界并发 → 订单状态乐观锁 + 幂等键互斥（非靠时间差缓冲；旧 29/30 双时钟方案作废）
用户通知 / 禁止重发站内信告知"资金已退回，请重新发起"；失败单不支持一键重发，避免重复扣款
4改造前 vs 改造后 · 效果对比
❌ 改造前痛点
  • 用户提交后立刻退出 APP → KYC 断连无人响应，订单作废
  • FASTUPI 单点订单和常规自动单混在一起，可能等几小时
  • 审单无 SLA，处理顺序随意，单点订单容易被遗漏
  • 断连提示只在"提现详情"页才看得到，用户切到其他页面就丢失
  • 失败处理无明确场景区分，资金停在中间状态，客服无法判断"退了没"
  • 用户重连成功 / 系统判失败 时间窗口冲突，可能出现重复出款或扣款
  • 客服无法快速定位订单卡在哪个环节
✅ 改造后效果
  • 提交时按银行类型主动提示用户保持在线 15-30 分钟
  • FASTUPI 优先派发：单点订单跳到队首，立刻入审单池顶部
  • 审单 15 分钟 SLA，超 10min 自动提醒审核员（不自动重派 / 不上报主管）
  • 客户端全局重连弹窗：任意页面实时弹出，附 29 分钟倒计时
  • 失败处理双分支明确：提交失败（KYC3005 · 29min 重连封顶）/ 提交成功（三方窗口 · 断连才弹），统一退回钱包
  • 单一服务端时钟 + 幂等锁：29min 只存服务端、客户端同步显示，临界并发靠状态锁防重复出款
  • 失败订单状态 = 已拒绝，资金原路退回 + 站内信通知用户重新发起
