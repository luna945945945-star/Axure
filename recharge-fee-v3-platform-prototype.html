二、字段定义名词术语

术语字段定义
下单金额Amount用户发起充值时提交的金额（充值发起金额）。仅用于限额判定与展示。
实付金额ActualAmount用户真实付款、三方真实到账的金额（原「实际充值金额」）。本次改造的统一业务基准——赠送、首充、VIP累计、累计充值、通道额度、打码基数全部以此为准。
手续费率ServiceFeeRate三方通道抽成比例，在商户+通道维度配置。存原始比率（0.02 = 2%）。0 表示该通道不收费。
费率快照FeeRate订单下单时固化的费率。之后后台改通道费率不影响历史订单，保证对账可复现。
手续费FeeAmount本单实收手续费 = 实付金额 × 费率快照。付给三方，平台并未收到。
上分金额CreditAmount= 实付金额 − 手续费。真正打进玩家余额的钱。仅用于：余额、账变、客户端「实际上分」。
充值赠送GiftAmount充值奖励/赠金，以实付金额为基数计算（不扣手续费）。与上分金额一并入账。
实际上分（客户端展示）客户端口径 = 充值金额 + 充值优惠 − 手续费，等价于 上分金额 + 充值赠送。
口径自检（充 ₹500 / 奖励 10% / 费率 2%）：赠送 = 500×10% = 50；手续费 = 500×2% = 10；上分金额 = 500−10 = 490；客户端「实际上分」= 500+50−10 = 540；余额 += 490+50 = 540。✅ 三处对齐。

三、业务规则

标签分类：口径 怎么算 · 展示 怎么显示 · 后端 怎么实现 · 边界 特殊情况。 测试主要看 展示 + 边界，开发主要看 口径 + 后端。
口径
BR-01 · 手续费率与快照
费率在「商户 + 通道」维度配置，每通道独立，0% 表示不收费；订单下单时固化费率快照
该配置项后台已存在且可录入，只是从未参与过计算，本期启用。之后调整通道费率不影响历史订单，保证对账可复现。
口径
BR-02 · 手续费与上分金额
手续费 = 实付金额 × 费率快照　｜　上分金额 = 实付金额 − 手续费
无手续费（0% / 本地通道 / 人工充值）时，上分金额 = 实付金额。汇总口径同理：上分金额 = 成功金额 − 手续费。
口径
BR-03 · 全局基准 = 实付金额　★ 本需求最核心的一条
赠送、首充判定与返佣、VIP / 充值等级累计、累计充值、通道剩余额度、打码基数 —— 一律按实付金额，不扣手续费
手续费只影响余额与账变两处，其余业务逻辑全部维持现状。例：充 ₹500、费率 2%，赠送仍按 500 算、首充阈值仍按 500 判 —— 不会因扣费后的 490 低于 500 而漏发首充。
后端
BR-04 · 上分存储过程改造
原「充值金额」入参继续传实付金额，新增一个「手续费」入参；仅「用户余额」与「账变」减去手续费
赠送 / 首充 / 返佣 / 通道额度的计算逻辑一行都不用改。存储过程内三处「余额累加」分支（首充统计表已存在 / 有其他成功充值 / 首充）均需减费，账变的入账金额与余额快照同步调整——漏改任一处 → 账变流水与余额对不上。打码量累加基数不变（仍按实付）。
展示
BR-05 · 展示通用约定
手续费金额为主 + 括号内费率；上分金额单列加粗突出；明细行无费率显示「—」，汇总行 / 卡片显示「0.00」
实付金额格内沿用中台标签堆叠样式，增加「手续费：」一行。两种空值展示不可混用。各页面的具体位置见右侧对应原型 Tab。
边界
BR-06 · 人工补单填实付
三方待入款 / 入款记录的「确认补单」，运营填实付金额，系统按费率快照自动算手续费与上分；不得填上分
反推实付需除以 (1−费率)，除不尽会产生精度误差且与三方对账不符；且只填上分会导致实付、手续费两列无数据来源。本地待入款 / 人工充值不走三方、无手续费，不受影响。
