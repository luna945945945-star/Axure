三种状态说明：
• 是 ↗ 该 UTR 在 KYC 流水中已找到 — 可点击 ↗ 跳到 KYC 流水查看匹配记录（验证金额、收款账号是否一致）
• 否 已填 UTR，但 KYC 流水中未找到 — 静态标签，不可点击（跳过去也是空结果，无意义）
• -- 没填 UTR 或非存款工单 — 不可点击
设计思路：判定只看"UTR 是否存在于 KYC 流水"，不做金额/账号的自动比对。套号判断交给客服——客服点击「是」跳过去后，肉眼对比工单金额 vs 流水金额，若不一致即为套号嫌疑。

性能说明：列表加载时一次性批量查询本页所有 UTR（IN 查询），不会 N+1 拖慢列表。需要确保 tab_kyc_transactionflowrecords 表的 (TenantId, Utr) 联合索引已存在。

① 会员id → KYC 流水（按 会员ID 检索）—— 查这个会员的所有流水

"原提款订单"卡片的关键设计：
• 异步加载：详情主体先打开，原订单信息异步从彩票系统拉取，避免详情打不开
• 失败降级：彩票 API 超时/失败 → 显示提示和"重试"按钮，不影响其他功能
• 出款通道是彩票系统真实出款的通道（非客户选择），与存款的"三方通道"概念不同
• 实到金额用红色高亮，便于客服快速发现"申请 vs 实到"差异
