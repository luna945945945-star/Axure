二、字段定义与名词术语
术语说明
充值大类 / 充值通道财务管理 → 充值配置 下的两级配置对象。大类按商户建，其下挂多条通道；通道是真正发起支付的对象。
充值大类奖励配在大类上的统一奖励比率，该大类下所有通道默认享有。
充值通道奖励配在单条通道上的奖励比率，用于对个别通道加码。
生效奖励max(充值大类奖励, 充值通道奖励)。实际发放给会员的比率。
跟随大类 / 跟随通道生效奖励的来源标识。通道奖励未配或不高于大类 → 跟随大类；高于大类 → 跟随通道。
角标客户端大类图标右上角的奖励标签，本次由单值改为区间值。
原始比率比率的存储与录入格式，四位小数，0.0200 = 2%。与 V1 通道的「充值赠送比率」、以及通道现有的「手续费」格式一致。
三、功能描述与业务规则
🎯 三个公式，先记这个
生效奖励 ＝ max( 充值大类奖励 , 充值通道奖励 )
角标区间 ＝ [ 充值大类奖励 , 该会员可见通道中的最高生效奖励 ]
实得赠送 ＝ 生效奖励 ＋ 快捷档位 ＋ VIP/等级奖励　（配了即叠加，不互斥）
下方 BR-01～BR-15 为细则；蓝色为核心规则，灰色为边界约定。
3.1 生效奖励算法 · 取高
BR-01取高生效核心：单笔充值的通道赠金 = max(充值大类奖励, 该充值通道奖励)。通道奖励高于大类则走通道，否则一律走大类。不存在「通道低于大类仍按通道发」的情况。
BR-02引用而非快照核心：通道对大类奖励是引用关系。大类比率变更后，所有跟随通道的生效奖励实时跟随，无需重新保存通道；原本高于旧大类值的通道，若新大类值反超，也自动改为跟随大类。
BR-03低于大类可存不生效：通道奖励低于大类时允许保存，仅不生效。保留该值的意义：日后大类比率调低后自动转为生效，运营可提前布局。

3.2 客户端 · 角标口径
BR-04角标口径核心：角标区间必须基于该会员当下真实可见可用的通道集合计算。
通道可见性条件是否计入角标
三端状态全启用✅ 计入
当前时间 ∈ 可用时间段✅ 计入
会员充值等级 ∈ 可见充值等级✅ 计入
轮询权重 > 0✅ 计入
充值金额 ∈ 充值金额区间❌ 不计入 —— 角标在用户输入金额之前就要展示，故忽略金额维度
违反本条（如把仅 VIP3 可见的 3% 通道算进普通会员角标）→ 会员看到「2%～3%」点进去只有 2% 的通道。这是本需求最主要的投诉风险点。
BR-05角标退化：区间上下限相等→ 显示单值；大类未配（仅通道有）→ 显示「最高 X%」；两级均未配 → 不展示角标。（四种形态的实际样式见原型「客户端 · 充值页角标」页签）
3.3 奖励结算 · 叠加
BR-06叠加口径核心：无独立叠加开关，配了即叠加、彼此不互斥。单笔充值的奖励结算顺序：
#奖励层配置位置关系
①通道赠金 = max(大类, 通道)本次新增取高，见 BR-01
②＋ 快捷金额档位赠送大类「快捷金额配置」现有命中该档位即叠加
③＋ VIP 等级 / 充值等级奖励大类「VIP充值赠送」开关现有已开启即叠加
④合计再与充值优惠券结算券的「通道赠金关系」互斥取高 / 叠加 / 互斥优先券
已核实 —— 叠加即 V1 现状，本次不引入「互斥取高」。 V1 实为四层全叠加、无任何互斥判断：①通道比例 ＋ ②快捷档位 在存储过程内相加（充值金额 × 通道比率 ＋ 快捷档位赠送额），合并为同一条「充值赠送」账变；③充值等级奖励 与 ④VIP 等级奖励各写自己的一条账变，由同一个方法串行调用、各自独立事务独立加余额，二者之间亦不互斥，各自独立封顶。
3.4 发放约定 · 均维持现状
编号规则说明
BR-07不设赠送上限本次不引入封顶（现状通道赠送亦无上限）。上限仅存在于充值等级 / VIP 等级两条独立线上。
BR-08打码倍数沿用 V1 现有的全站「充值赠送打码量倍数」设置，不新增大类级 / 通道级字段。打码量 = 全站倍数 × 赠送总额。
BR-09延迟发放仍为通道级、仍在 V1 后台配置（充值通道列表 →「常用修改」→「延迟发放时间」），V3 中台无此字段。0 = 即时到账，N = N 天后零点由定时任务派发。
BR-10比率变更的
订单口径		已下单未提交三方按原比率，新发起的按新比率。
⚠ 需新增字段：现状充值订单没有比率快照，奖励是到账回调时现读配置算的（即当前行为 = 按到账时的最新比率）。须在下单时把生效比率快照下来，到账时读快照。
3.5 录入 · 防护 · 权限
BR-11修改防护 =「是否修改」校验闸门核心：比率属资金字段，完全沿用 V1 既有防护机制。
要素规则
是否修改只是校验闸门，不是业务开关。不勾选时，提交后该字段的新值一律被忽略（原值不动）—— 防止编辑其它字段时误改比率。
谷歌效验码勾选后必须输入 6 位方可提交，校验失败不予保存。
作用域大类 = 仅「充值赠送比率」（与 V1 一致）；通道 = 「奖励来源 ＋ 比率」。「快捷金额配置」不加保护（V1 该页亦无此字段）。
V1 现有实现（后台「充值设置」页）：勾选「是否修改」后，比率输入框解除只读，同时置上一个「需校验谷歌码」的标记提交给后端，由后端据此决定是否校验。
为何通道也必须保护：通道奖励双写的正是 V1 通道的「充值赠送比率」—— 即 V1「是否修改」原本保护的那个字段。V3 侧若去掉，会造成「同一个字段，V1 改要谷歌码、V3 改不要」的绕过口子，属防护降级。
编号规则说明
BR-12比率格式存储/录入用原始值、展示用百分比：存 0.0200，列表列与客户端角标显示 2%（＝ 原始值 × 100）—— 与现有「手续费」列 0.9000 显示为 90% 的口径一致。
BR-13不设独立
「奖励开关」		是否发奖励由比率值本身表达：0 = 不配，> 0 = 生效。理由：① V1 本就没有这个开关，比率填 0 即无优惠；② 到账计算是「充值金额 × 比率」，比率 0 天然不发；③「开关关闭」与「比率填 0」是同一个业务状态，两个控件表达一个状态会造成配置歧义。
BR-14权限与审计不新增独立权限码，有充值大类编辑权限即可改；变更记操作日志（操作人 / 时间 / 原值 → 新值），复用列表「最后修改人 / 最后修改时间」。
BR-15配置粒度商户级（大类按商户建，通道按商户绑定），各商户独立，无跨商户统一配置。
四、限制条件
项说明
比率校验非负数、四位小数；0 为合法值（= 不发奖励），无「必填且不可为 0」限制；通道选「自定义」时须 > 0。
准入过滤优先通道的「优惠可用等级」「优惠可用设备」白名单不匹配时，该会员在该通道不享受任何奖励（比率清零）。此规则优先于本次取高逻辑，且角标计算须一并遵守。
本次不做赠送上限、大类/通道级打码倍数、大类级延迟发放天数、奖励活动有效期、按会员分层的差异化奖励、奖励层之间的互斥取高。
